Protocol Edge Calibration

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Calibrate the answerability benchmark and coherence checks so shipped route-node and http_calls evidence is treated as passing protocol behavior. The goal is to stop stale route-capability expectations from driving a protocol-roadmap recommendation while preserving honest failure signals for real missing evidence.

Design decisions (spec was ambiguous → here's the call I made)

Decision Use the existing benchmark harness as the public interface. The PRD is about answerability calibration, not extractor internals, so tests will exercise scenario scoring, report summaries, and coherence checks through existing eval entry points rather than private route extraction helpers.
Decision Keep #419-#421 as the canonical child issues. Four duplicate issues were accidentally created during concurrent publishing (#422-#425). They were closed as duplicates, and implementation will target the already-present vertical slices #419, #420, and #421.
Decision Use per-scenario output budgets for known bounded protocol answers. The route-caller scenario returned complete facts and evidence but exceeded the global 500-token default by a small amount. A scenario-local budget keeps the default strict for most cases while avoiding a false protocol-extraction failure for a verified graph-search answer.
Decision Keep protocol coherence schema-based in this slice. The implemented guard compares scenario capability expectations with the fixture schema. Graph-search evidence coherence is intentionally not part of this pass because the stale failure being fixed is a schema-capability contradiction, and adding graph-search replay would broaden the scope beyond calibration.

Deviations (intentional departures from the spec/convention)

Deviation No new HTTP extractor scope. The live worktree schema reports populated route nodes and http_calls evidence, so this implementation deliberately avoids rebuilding extraction and focuses on stale scenario expectations, failure classification, and coherence protection.

Tradeoffs (alternatives considered)

Tradeoff Positive route scenario vs. unsupported-honest route scenario. Keeping the old unsupported route scenario would preserve a deferral signal but make the benchmark contradict the fixture. Updating it to positive route evidence gives future agents a truthful regression guard for shipped protocol support.

Open questions (want your confirm/revise)

Open None currently blocking. The implementation assumes protocol calibration should pass the current fixture and move the roadmap recommendation away from protocol unless another non-passing protocol scenario remains.
