The spec, in one line

Make Seam's answerability benchmark, docs, capability language, and roadmap signals separate real failing agent questions from deferred roadmap pressure and stale metadata.

Design decisions (spec was ambiguous → here's the call I made)

Decision Keep coherence under eval support first. The PRD asks for graph-quality coherence, but the concrete current drift is in the answerability benchmark/report path. The implementation keeps coherence in deterministic eval modules and leaves product CLI surfaces unchanged because no test required a public command.
Decision Legacy product_gap_tags compatibility. Existing scenario records may still use product_gap_tags. For compatibility, those tags now count as failure gaps only when a scenario is non-passing; if the scenario passes, they count as roadmap pressure so old metadata cannot dominate the failure recommendation.
Decision Regression coverage fails closed. Review found that a non-passing scenario with only regression_tags could stay classified as coverage. The classifier now treats regression tags as coverage only while the scenario passes; when it fails, the tag becomes a failure gap and can drive the recommendation.

Deviations (intentional departures from the spec/convention)

Deviation No interactive issue-breakdown gate. The to-issues skill normally asks for approval before publishing slice issues. The user explicitly requested issue creation and full implementation in the same turn, so I published four AFK slices (#413-#416) and proceeded.

Tradeoffs (alternatives considered)

Tradeoff New metadata vs. reusing product_gap_tags. Reusing the old field would be smaller, but it preserves the ambiguity that caused stale roadmap pressure. The implementation migrates to explicit failure/pressure/regression semantics while keeping a compatibility path for older scenario records.
Tradeoff Coherence checks live in eval support. A product CLI command would be more discoverable, but the current need is PR-time deterministic validation. Keeping the module under tests/eval avoids expanding Seam's shipped MCP surface while still making the check available through answerability_report --coherence.

Open questions (want your confirm/revise)

Open Tracker audit application policy. The PRD says optional tracker audit should not mutate by default. I implemented dry-run classification only; issue closing at the end of this implementation will be limited to the four slice issues created for this PR.
