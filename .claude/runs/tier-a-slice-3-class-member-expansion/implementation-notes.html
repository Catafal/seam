Tier A Slice 3: class->member expansion in context() + query()

Branch feat/tier-a-name-resolution · 2026-06-06 · Slice 3 of 4 in the Tier A read-path name-resolution fix.

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Companion to Slices 1+2 which are already committed.

The spec, in one line

When a query name is a class/struct/interface, fan into its member symbols (qualified_name LIKE 'Class.%') so edge_match_names includes the members' bare names, causing their callers to union up to the class. Bounded by a new SEAM_NAME_EXPANSION_CAP config knob (default 50). Zero-member classes degrade gracefully. Wire into context() and query() 1-hop expansion.

Design decisions

Decision is_container_symbol() checks kind column, not a naming heuristic. The spec says "class/struct/interface symbol" — rather than guessing from the name (e.g. PascalCase), we query the kind column in the DB. The closed vocabulary is {"class", "interface", "struct"}. This is exact: Go structs, C/C++ structs, TS interfaces, Python classes all have correct kinds set by the extractors. A method named "Parser" would not trigger fan-out because its kind is "method", not "class".
Decision get_member_names() uses LIKE 'Class.%' with Python-side prefix confirmation. Same pattern as the bare-suffix scan in Slice 2. LIKE is the SQL pre-filter; a Python startswith() check ensures the prefix is exact. This prevents a class named "Foo" from matching "FooBar.method" (the LIKE pattern "Foo.%" won't match "FooBar.method" anyway, but the guard is kept for consistency with the existing naming conventions in the module).
Decision edge_match_names() extended: container bare name fans out to member names. A class symbol like "CompanionManager" has no call edges targeting "CompanionManager" directly (callers invoke "start", "stop" — the bare method names). To make context(class) see callers of members, edge_match_names must return ["CompanionManager", "start", "stop", ...] when the name is a container. Non-containers and qualified names (with a dot) are unchanged from Slice 1.
Decision query() 1-hop expansion reuses edge_match_names per seed, not a flat seed list. The existing query() neighbor SQL used a flat IN (seed_names) clause. For class seeds this missed all callers-of-members (no edge targets the class name itself). The fix: for each seed name, collect edge_match_names(conn, seed_name) and union them into a single edge_lookup_names list, deduped. The SQL shape is unchanged; only the parameter list grows.

Deviations

Deviation No separate "container outline" in context() result (Tier D item 11). The spec for Slice 3 only requires aggregating callers via member fan-out, not adding a member list to the ContextResult dict. Adding a members field would require a TypedDict change touching all callers. Kept out of scope per the "read-path only, minimal surface" principle. Tier D item 11 (container outline) is the natural next step but is a separate slice.

Tradeoffs

Tradeoff LIKE-based member scan vs. a DB index on the prefix. A column-prefix index on symbols.name would make LIKE 'Class.%' fast even on huge indexes. The existing idx_symbols_name is on the full name and SQLite CAN use it for prefix LIKE (it's a B-tree scan from "Class." to "Class.\xff"). On realistic codebases (<100k symbols) this is negligible. The cap (50+10 buffer) bounds the worst case. A separate "members" table or a "contains" edge would be the durable fix (Tier B item 4).
Tradeoff Checking is_container_symbol() in edge_match_names() vs. caching the result. Each call to edge_match_names() does up to 2 DB queries (is_container + get_members). For context(), this is called O(N defs) — typically 1 call. For query() neighbor expansion, it's called once per seed (O(seeds), typically <10). Caching across calls would add complexity; the cost is a handful of indexed lookups per query, which is negligible vs. the FTS5 and edge queries already in the path.

Open questions

Open Should "type" kind (TypeScript type aliases, Rust type aliases) be a container? TypeScript type Foo = {...} is stored as kind="type". In practice, type aliases don't have member methods, so fan-out would return []. Including "type" in _CONTAINER_KINDS would add one extra DB query per "type" symbol for zero benefit. Left out of _CONTAINER_KINDS for now. Revisit if type aliases with method-like members appear (unlikely in current extractors).
