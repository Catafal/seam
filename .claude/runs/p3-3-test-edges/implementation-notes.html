P3.3 Test Edges

Branch codex/p3-3-test-edges · started 2026-07-02 · implements PRD #163.

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Materialize conservative tests edges from test symbols to production symbols using static evidence, then expose those edges through schema, graph search, context, architecture, Explorer, and docs without changing seam_affected output shape or default production impact semantics.

Design decisions (spec was ambiguous → here's the call I made)

Decision Use ordinary graph edges, not a new test metadata table. The PRD allowed a metadata table if richer evidence required it. The first pass can encode source, target, kind, confidence, and provenance through the existing edge model, so avoiding a table keeps P3.3 additive and old-index friendly.
Decision Exact evidence before proximity. The build starts with call, instantiation, and import evidence because those relationships already exist in the graph. Name-matched import evidence is intentionally narrower than file-level imports, and proximity heuristics stay lower confidence.
Decision Materialize as a whole-index post-pass. Test edges need to know whether an edge target exists in production code, which is not knowable from one file's AST alone. The implementation therefore runs after file indexing and derives tests edges from the complete symbol/edge tables.

Deviations (intentional departures from the spec/convention)

Deviation No runtime coverage integration. This follows the PRD's static-evidence boundary. Indexing will not run tests or parse coverage reports, so tests means "static relationship evidence" rather than coverage proof.
Deviation Use synthesized_by for test-edge provenance. The generic edge table has no separate provenance column. Because P3.3 edges are derived by a post-pass, synthesized_by carries channels such as test-call and keeps these edges out of clustering and default impact paths.

Tradeoffs (alternatives considered)

Tradeoff Materialize during indexing vs. infer at read time. Read-time inference would avoid writing more edges but would duplicate work across tools and make Explorer/architecture harder to keep consistent. Materialization follows the route/config/exception pattern and gives every read surface the same evidence.

Open questions (want your confirm/revise)

Decision Ship proximity, but only for unique targets. A proximity edge is useful when test_parse_config has exactly one production parse_config target. Ambiguous or absent targets are skipped, so the signal remains discovery-grade rather than pretending to be coverage.
