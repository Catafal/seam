Tier A Slice 1 — Name Resolver Module

Branch feat/tier-a-name-resolution · started 2026-06-06 · qualified<->bare name bridging in context() to fix empty callers/callees for Class.method symbols.

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Create seam/query/names.py with bare_name() and edge_match_names() as a stdlib+config-only leaf module, then wire edge_match_names into engine._build_context_result so that context("Class.method") returns real cross-class callers that were previously empty, while unique-name functions remain byte-identical.

Design decisions (spec was ambiguous -> here's the call I made)

Decision conn param is threaded through but not used yet. The spec says edge_match_names(conn, name) takes a conn. For Slice 1 no DB query is needed — the bare name is just a string operation. I kept the param for API stability: Slice 3 (member expansion) will need to query SELECT DISTINCT name FROM symbols WHERE qualified_name LIKE 'Class.%' and callers must not change their call site when that lands.
Decision DISTINCT in caller/callee SQL queries. The original queries used SELECT source_name FROM edges WHERE target_name = ? (no DISTINCT). After the IN-clause expansion to two names, it is possible for both "Class.method" and "method" to match the same edge row — which would duplicate the caller in the list. Adding DISTINCT to the SQL ensures deduplication happens at the DB level (cheaper than Python-side dedup) and is safe for the unique-name case too (DISTINCT on a single match is a no-op). This satisfies TA4 (dedup test).
Decision Order in edge_match_names: qualified first, bare second. The spec says "at minimum the qualified name itself PLUS its bare suffix" but does not specify order. I chose qualified-first so the first element is always the name the caller asked about — useful for debug logging and future differential logic (e.g. "did this match come from the exact name or the bridged bare form?").

Deviations (intentional departures from the spec/convention)

Deviation DISTINCT added to caller/callee queries (not in spec). The spec says wire edge_match_names into the IN clause. It does not mention DISTINCT. I added it because without it the dedup test (TA4) would fail when a caller has BOTH a bare and a qualified edge to the same method. This is strictly safer and does not change results for any pre-existing case.

Tradeoffs (alternatives considered)

Tradeoff IN-clause expansion vs. Python-side union. Alternative: run two separate queries (WHERE target_name = qualified AND WHERE target_name = bare) and union in Python. Chosen: single IN-clause query. Reason: one round-trip is cheaper than two; DISTINCT is expressed in SQL; and the approach naturally extends to N names (Slice 3 may add more members). Downside: slightly harder to explain, but the comment in the code makes the intent clear.
Tradeoff Leaf module vs. inline helpers in engine.py. The spec mandated a separate seam/query/names.py. Alternative would be to inline bare_name and edge_match_names in engine.py. The leaf module is strictly better: testable in isolation, mirrors the clusters.py pattern, and keeps engine.py under its line limit.

Open questions (want your confirm/revise)

Open Should callees also use the bare source matching? The spec says "callers/callees via edge_match_names". I applied the IN-clause expansion to BOTH callers and callees. For callees, source_name is always stored as the qualified name (graph_common.py uses the enclosing qualified name), so the bare match adds no new results in practice — but it is safe and future-proofs the case where source_name is ever stored bare. If you want callees to match only the exact qualified name, the fix is one line.
