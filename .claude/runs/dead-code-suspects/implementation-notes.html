The spec, in one line

Add a conservative read-only suspect-analysis surface that helps agents review weakly connected symbols and files without ever certifying that code is dead or safe to delete.

Design decisions (spec was ambiguous → here's the call I made)

Decision Five AFK vertical slices. The issue split keeps the core classifier, file/target modes, transport parity, answerability, and docs/review independently verifiable while still shipping one coherent product surface.
Decision Separate suspect layer. The existing graph-search recipes remain low-level structural discovery. A dedicated seam_suspects layer owns cleanup-specific language, blockers, removal risk, caveats, and follow-up calls so agents cannot confuse a degree filter with a deletion recommendation.

Deviations (intentional departures from the spec/convention)

Deviation Review checklist unavailable. The requested /review skill requires .claude/skills/review/checklist.md, but that file is not present in this worktree. I am proceeding with the loaded backend-taste, QA, test, and diff review checks instead of blocking the build.

Tradeoffs (alternatives considered)

Tradeoff Suspect surface vs. graph-search-only. The raw `dead-code-suspects` recipe already exists, but PRD #364 needs a richer layer because cleanup review depends on blockers and caveats, not only degree filters.
Tradeoff Static proof is intentionally impossible. Candidates can be strong, moderate, or weak, but never safe-to-delete. That keeps dynamic loading, external consumers, generated code, and framework conventions visible as caveats rather than hiding them behind a false binary result.

Open questions (want your confirm/revise)

Open Repository cleanup reports. This ships an agent-readable, bounded review tool. If product usage later needs a full repository cleanup report, that should be a separate explicit artifact mode with pagination/export semantics.
