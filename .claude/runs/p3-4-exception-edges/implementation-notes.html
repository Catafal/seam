P3.4 Exception Edges

Branch codex/p3-4-exception-edges · started 2026-07-01 · implements PRD #155 and child issues #156-#160.

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec.

The spec, in one line

Add conservative raises and catches graph edges for explicit Python and JS/TS exception evidence, then expose them through existing Seam schema, graph-search, architecture, web, Explorer, and documentation surfaces without runtime probing.

Design decisions (spec was ambiguous → here's the call I made)

Decision Store exception evidence as normal edges. The existing edge contract already carries kind, target, file, line, and confidence, so no exception metadata table or schema migration was needed for P3.4.
Decision Attach only to named scopes. The extractor uses the same enclosing-symbol resolver as call edges; module-level exception syntax is skipped because Seam has no module symbol row to target without inventing a new node model.
Decision Architecture includes exceptions by default. The section is bounded and compact, so the default repo briefing now surfaces raised types, caught types, broad catches, and raise/catch-heavy symbols.

Deviations (intentional departures from the spec/convention)

Deviation Typed catch extraction for JS/TS starts as unsupported. Ordinary JavaScript and TypeScript catch clauses do not reliably expose concrete exception types, so P3.4 skips untyped catches instead of guessing from variable names.

Tradeoffs (alternatives considered)

Tradeoff Deep extractor module vs. extending each graph extractor. A standalone exceptions extractor avoids mixing failure-flow rules into call/import extractors and keeps unit tests focused on public extraction behavior.
Tradeoff Explicit evidence over propagation. Exception propagation through callees would be tempting for impact analysis, but this phase keeps the graph trustworthy by indexing only visible raise/throw/catch syntax.
Tradeoff Keep default impact focused on direct change risk. seam_impact excludes raises/catches by default so exception syntax is available for failure-path review without inflating blast-radius tiers.

Open questions (want your confirm/revise)

Open Future propagation semantics. Full exception propagation through call graphs remains out of scope and can be layered later if explicit edges prove valuable but insufficient.
