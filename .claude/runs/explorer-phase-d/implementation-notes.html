Explorer Phase D — Implementation Notes

Branch explorer-phase-d · 2026-07-03 · Closes the linear A→D Explorer redesign arc (#255, slices #272–#275).

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The phase, in one paragraph

Phase D closes three UX gaps that remained after the A→C arc: (1) the tab system relabelled itself with the OTHER mode's name (anti-pattern killed in D2 by an explicit TabBar with fixed labels and a single-source lib/tabs.ts); (2) the header carried operational index stats, consuming space that belongs to navigation (demoted to a bottom StatusStrip in D3, backed by the D1 additive stale/stale_reason fields on GET /api/status); (3) navigating from a landing area card into a symbol's neighborhood and then clicking a node left the user with no breadcrumb back (fixed in D4 by a pure deriveCrumbs() function and a thin Breadcrumb component spanning every surface). One additive backend change: stale: bool + stale_reason: str | None on /api/status, derived from the same check_staleness(respect_knob=False) call used by the MCP graph-traversal handlers so the three staleness signals can never disagree.

Design decisions (spec was ambiguous → here's the call I made)

Decision StatusResponse moved to web_schema.py. web.py was at 991 lines. Adding the two new fields + the staleness import + the staleness call in the handler would have pushed it past 1000. The established pattern (see StructureSymbol, SchemaResponse, SnippetResponse) is to extract Pydantic models into web_schema.py when the 1000-line ceiling approaches. StatusResponse was the most isolated model (no forward references, no other models depend on it), so it was the right candidate. web.py ends at 993 lines; web_schema.py at 424 — both comfortably under 1000.
Decision stale_reason is None when stale=False. The spec says stale_reason: str | None but does not state what the reason field should be when the index is fresh. Mirroring the existing MCP banner contract (index_status is absent entirely when fresh, not present with empty reason), I set stale_reason=None when verdict["stale"] is False. The handler-common _maybe_attach_staleness sets reason only when stale; this is the same semantic. Fresh = no reason = None.

Deviations (intentional departures from the spec/convention)

No deviation Followed the spec and existing patterns exactly. The staleness call mirrors handler_common._maybe_attach_staleness line-for-line (same pid_file derivation, same _watcher_is_alive call, same check_staleness call with identical args). No re-implementation.

Tradeoffs (alternatives considered)

Tradeoff Call staleness inside vs. outside the try/finally block. The staleness check needs the open conn (it queries files for indexed_at). Placing it inside the try block keeps it inside the connection lifetime; outside would require re-opening the connection or restructuring. Chose inside: one connection, one finally-close, same pattern as every other route handler in the file. Consequence: if the staleness check raises (it never does — it's documented to degrade gracefully), the connection is still closed correctly by the finally block.

Open questions

None No open questions for this slice. The spec was unambiguous, the source of truth (check_staleness) was already established, and the model-extraction pattern was precedented. This is a purely mechanical additive change.

D2 (#273) — Explicit Overview/Symbol/Topology tab bar

Decision ViewMode type moved to lib/tabs.ts; imported from there in App.tsx. The single source of truth for the tab list and the ViewMode type is now lib/tabs.ts. App.tsx imports both TabBar (the component) and ViewMode (the type) from their respective new homes. The old local type ViewMode = ... declaration in App.tsx was removed. This keeps the label↔mode mapping co-located with the type definition so they cannot drift apart.
Decision HeaderToggle removed entirely; Changes drawer keeps a plain button. HeaderToggle was the component that relabelled itself with the OTHER mode's name (the anti-pattern). It was only used for two things: the Overview⇄Neighborhood toggle and the Changes drawer toggle. Both are replaced inline. The Overview⇄Neighborhood toggle is now the TabBar. The Changes button is now a plain inline button with aria-pressed (it's a toggle, not a navigation tab — it does not switch the main content area). This avoids HeaderToggle surviving as dead code, and keeps Changes visually distinct from the three navigation tabs (restraint principle: one sky-accent element = the active tab).
Decision iconName as a string discriminant in TabDef, resolved in TabBar. tabs.ts must be importable in Node.js test environments without React. Storing ReactNode or a Lucide component reference in TabDef would create a React/DOM dep in a pure-data module. Instead, iconName is a string union "network" | "search" | "orbit" resolved to the actual Lucide component by a ICON_MAP constant in TabBar.tsx. The pure helper stays truly pure; the component owns the React-specific mapping.
Test update TopologyToggle.test.tsx selectors changed from role="button" to role="tab". The existing tests used getByRole("button", { name: /topology/i }). After the change the Topology item carries role="tab" (from the TabBar), so the role was updated to "tab" across all existing test files. Additionally, the "deactivating Topology" test was updated to click the Symbol tab instead of clicking the already-active Topology tab (explicit tab bars do not toggle on re-click — that was the old toggle-pill behavior being killed).
Tradeoff aria-current="page" vs aria-selected on tabs. The full ARIA tabs pattern (role="tablist" + role="tab" + role="tabpanel" with aria-selected and focus management) was considered. We use a simplified form: role="tablist" + role="tab" + aria-current="page". Reason: the "panels" are the entire main-content area (not DOM siblings of the nav), so aria-controls cannot point to a specific element. aria-current="page" is semantically correct for a navigation landmark that marks the current destination. Keyboard focus management (arrow key navigation within the tablist) is out of scope for this slice — the TabBar is not a complex widget, just three clearly labelled navigation buttons.

D3 (#274) — Bottom server-admin status strip

Decision StatusBadge removed entirely from the header; StatusStrip is the sole owner of index stats. The spec says "move the existing StatusBadge + formatRelative logic OUT of the header into it". Removed StatusBadge function and formatRelative utility from App.tsx; useStatus import also removed from App (StatusStrip calls it directly). This means the header has zero awareness of index status, matching the "header keeps navigation only" spec requirement. formatRelative lives in StatusStrip.tsx now — single owner, single location.
Decision No-layout-shift via a blank spacer, not invisible text. The spec says "no layout shift when the stale state toggles (reserve space)". First attempt used visibility:hidden on the stale text — but queryByText(/stale/i) in tests still found the hidden text, making the "shows NO stale warning text when stale=false" test fail. Fix: render an empty <span> with h-4 w-48 in the fresh branch (same pixel dimensions as the stale indicator) — reserves layout space with zero text content. Tests are now stable; queryByText(/stale/) returns null for fresh state as expected.
Decision data-testid="stale-indicator" as stable test hook for amber state. Tailwind class names are implementation details; they can change in refactors without breaking the semantic intent. The test asserts the presence of data-testid="stale-indicator" (not a class or color) to verify the amber warning element exists. This follows the established pattern seen in other Explorer tests.
Decision types.ts updated manually (not regenerated from OpenAPI). The generated types.ts did not yet include the D1 stale and stale_reason fields (regeneration would require running the FastAPI schema endpoint and the codegen pipeline — not possible here). Added them manually with JSDoc matching the Pydantic model docstring. The StatusResponse shape in web_schema.py is authoritative; types.ts is kept in sync here by manual update.
Tradeoff amber as role="alert" aria-live="assertive". The stale warning is the only time this strip carries actionable urgency. Using role="alert" and aria-live="assertive" on the stale indicator means screen readers announce it immediately when it becomes visible. The fresh spacer element carries aria-hidden="true" so it is invisible to assistive technology. Tradeoff: assertive can interrupt ongoing speech on screen readers. Given the stale warning is infrequent (fires only after a real index drift is detected) and actionable (names the exact fix), this is the right balance — the user SHOULD be interrupted.
Decision StatusStrip is always rendered (not gated on mode or symbol state). The strip is operational metadata, not content — it belongs at the app shell level alongside the header. It renders in all modes (neighborhood, overview, topology) and whether or not a center symbol is selected. This means one useStatus() call, one render path, no conditional mounting that would cause the strip to flash in/out during navigation.

S4 — End-to-end breadcrumbs (#275)

Decision Pure deriveCrumbs(state, handlers) in breadcrumbs.ts. The spec called for a "pure state → Crumb[] derivation" — but crumbs need click handlers, which are side-effecting closures. Resolution: inject the handlers as a second argument, keeping the function deterministic (same inputs → same outputs) while remaining fully unit-testable with vi.fn() mocks. This pattern mirrors how resolveClusterHandoff in lib/ is pure + injectable.
Decision Mode-awareness in deriveCrumbs (topology = repo only). The PRD listed the trail as "repo → area → symbol → selected" without mentioning mode, but including mode in BreadcrumbState is the right call: in topology mode, centerSymbol may still be set from prior navigation, but the topology surface has no drill path — showing a symbol crumb there would be misleading. Topology clamps to [repo]; overview shows up to [repo, area]; neighborhood shows the full chain. Tests cover all mode × state combinations.
Decision Selected-node crumb always has an onClick (even if it's a no-op). The PRD says "Crumbs are keyboard-focusable (buttons)." A <button> with no onClick is still focusable, but for semantic consistency all crumbs carry an onClick. The selected-node crumb (the deepest level, always isCurrent=true) has onClick: () => {} — a harmless no-op. This avoids special-casing in the renderer and keeps the Crumb type uniform.
Decision <main> restructured to flex-col with breadcrumb at row 1. The breadcrumb must span ALL surfaces. The cleanest structural change: make <main> a flex-col (was flex / horizontal row) with breadcrumb as shrink-0 row 1 and a new wrapper <div class="flex flex-1 overflow-hidden"> as row 2 containing the existing horizontal surface layout. No existing component was touched; the inner flex-row structure is unchanged.
Decision openCenterSymbol clears selectedSymbol + forces neighborhood mode. The symbol breadcrumb's job is to restore the center-symbol-only view when a deeper selectedSymbol exists. Setting selectedSymbol(null) and setMode("neighborhood") is the minimal restoration. centerSymbol is untouched (it's already the target level). This mirrors how clicking "✕ clear" in the header resets both, except here we keep the center.
Trade-off No URL-based routing (in scope for this slice). Breadcrumbs are in-memory state. Deep-linking is called out as "out of scope" in the PRD and is left for a future item. The useMemo on crumbs avoids re-creating onClick closures on unrelated state changes (e.g., changesOpen), but the gain is correctness, not a performance optimization.

Review Fix — respect_knob=False on /api/status (#255)

Bug Fix check_staleness at line 601 changed from default respect_knob=True to explicit respect_knob=False.

The original call used the default respect_knob=True. When SEAM_STALENESS_CHECK=off, check_staleness short-circuits and returns stale=False unconditionally (see staleness.py:246). Meanwhile, both seam/query/schema.py and seam/query/architecture.py call check_staleness with respect_knob=False, so they return the real staleness verdict regardless of the knob. This caused /api/status to disagree with /api/schema and /api/architecture — precisely the invariant PRD user story 16 prohibits ("cannot disagree with seam status").

Fix is a one-liner: respect_knob=False on the check_staleness call in the /api/status route handler. The existing comment already claimed this endpoint "cannot disagree with seam status" — the fix makes that claim true even when SEAM_STALENESS_CHECK=off.

The integration test test_status_stale_fields_present exercises a freshly-indexed fixture (always fresh), so it does not exercise the SEAM_STALENESS_CHECK=off divergence path. The fix is correct by code inspection: respect_knob=False bypasses the knob entirely, identical to what every other endpoint and seam status CLI does.
