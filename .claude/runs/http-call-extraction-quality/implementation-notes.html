HTTP Call Extraction Quality

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Improve Seam's conservative static HTTP-call extraction so agents can answer "who calls this route?" through existing graph/search/schema/architecture surfaces, without runtime probing, semantic dependency inference, or false local route edges.

Design decisions (spec was ambiguous → here's the call I made)

Decision Use existing route symbols and edge rows. The PRD allows a new deep matcher but does not require storage changes. The implementation will first keep matched local calls as ordinary http_calls edges to route symbols so schema, graph search, architecture, workspace federation, and artifacts continue to consume one graph model.
Decision Start with supported wrapper names, not arbitrary data flow. Local wrapper recognition is valuable for Seam's own apiFetch usage, but arbitrary function inference would create false protocol edges. The first pass recognizes supported wrapper names such as apiFetch and direct library calls only when method/path evidence is static.
Decision Keep unresolved same-repo-looking targets as existing compatibility for now. Current Seam tests and graph-search previews already support route_resolved:false for literal HTTP calls whose route target is not indexed. This build adds resolved wrapper/Python coverage first and will harden the wording/tests around resolved route-caller evidence rather than silently removing the existing unresolved edge behavior mid-slice.
Decision Scope Python client receivers to the enclosing symbol. A review pass found that file-global receiver tracking could leak a local session from one function into another. The implementation now combines module-level receivers only when a function does not shadow that name, and keeps local client receivers within the function/method that declares them.

Deviations (intentional departures from the spec/convention)

Deviation No unresolved external-call inventory in this PR. The PRD permits skipping external/dynamic calls. Keeping them out of storage preserves the current graph contract and avoids designing an external dependency model inside this route-caller quality slice.

Tradeoffs (alternatives considered)

Tradeoff False negatives over broader matching. A more aggressive extractor could catch more wrapper/base-path cases, but wrong route edges damage impact, architecture, and agent trust. This build will prefer exact or narrowly near-literal evidence with product-facing provenance.

Resolved questions

Resolved Impact traversal policy already includes http_calls. Current impact defaults exclude raises, catches, and tests; protocol edges remain part of production graph traversal. This PR leaves that policy unchanged and documents that dynamic/external HTTP calls are omitted instead of added as speculative graph edges.
