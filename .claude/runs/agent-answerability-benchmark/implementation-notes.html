The spec, in one line

Build a deterministic, local-first benchmark that asks natural-language coding-agent questions, runs current Seam read tools, scores answerability/evidence/caveats/cost, and reports which roadmap bucket should be prioritized next.

Design decisions (spec was ambiguous → here's the call I made)

Decision Benchmark lives under eval tooling, not product CLI. The PRD says this is an evaluation surface rather than new graph extraction. I followed the existing repo convention by adding tests/eval/answerability_* and make eval-answerability, leaving the runtime seam CLI unchanged.
Decision Scenario truth is authored JSON. The maintained catalog is structured data in answerability_scenarios.json because future agents need to review question intent and expected evidence without reading Python.
Decision Unsupported capability can still be a passing answer. Protocol and infra scenarios pass when Seam reports unsupported/empty evidence with an accepted caveat. This prevents the benchmark from rewarding false certainty.

Deviations (intentional departures from the spec/convention)

Deviation No live agent or LLM judge in this phase. The PRD allowed optional LLM judging later. I kept this first implementation deterministic so it can run in normal local development without network, credentials, or reviewer variance.
Deviation Optional local repos are deferred. The PRD permits optional external repos that skip cleanly. I kept the first implementation on the deterministic eval fixture so the benchmark has a stable base before adding machine-specific paths.

Tradeoffs (alternatives considered)

Tradeoff Fixture-sized truth vs. broad realism. A 20-scenario fixture suite is less realistic than running across multiple private repos, but it gives a stable regression base. Larger repo scenarios can be added once the scoring/report shape is proven.
Tradeoff Simple evidence normalization over per-tool bespoke parsers. The normalizer walks JSON-like payloads for common evidence keys. This is deliberately smaller than a custom parser per Seam tool, while still supporting the current scenario set.

Open questions (want your confirm/revise)

Open Which non-Seam repos should become optional scenarios? The harness can grow to optional local repos, but the first implementation does not choose private repo paths. Adding them later is a scenario-catalog and adapter extension, not a redesign.
