Semantic Discovery Productization

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Productize Seam's existing opt-in semantic search as a safe discovery surface: agents must see semantic readiness, retrieval mode, caveats, and follow-up guidance without letting similarity affect dependency, risk, trace, graph-degree, suspect, or docs-grounding semantics.

Design decisions (spec was ambiguous → here's the call I made)

Decision Keep semantic productization additive. Existing CLI/MCP callers already consume search/query payloads, so this implementation will add optional readiness and retrieval metadata instead of replacing current result fields.
Decision Use synthetic/vector fixture tests before any real model dependency. The PRD requires no query-time network and no default model download; tests will use existing fixture patterns and synthetic embeddings before any live semantic benchmark path.
Decision Centralize semantic vocabulary. seam/query/semantic_contract.py owns readiness reasons, retrieval modes, caveats, and recommended next calls so schema, engine, MCP, Web, docs, and tests do not drift.

Deviations (intentional departures from the spec/convention)

Deviation Answerability scenarios validate the stable contract, not live semantic recall. The maintained answerability fixture does not build embeddings, so the shipped scenarios check readiness honesty and keyword-only retrieval metadata. Synthetic embedding tests cover semantic-only and hybrid recall directly.

Tradeoffs (alternatives considered)

Tradeoff Per-result metadata vs. response-level status. Per-result retrieval mode is necessary for semantic-only and hybrid leads, but readiness and fallback state should be response-level to avoid bloating every result.
Tradeoff Schema readiness reuses the optional-extra check. This gives agents the truth before search, but it may import-check the semantic extra during schema/status calls. It does not load a model or embed a query.

Open questions (want your confirm/revise)

Open None remaining for this implementation. Future semantic benchmark work can add a fixture that builds synthetic embeddings through the eval harness, but the PRD scope is covered without adding model/network dependencies to the maintained answerability run.
