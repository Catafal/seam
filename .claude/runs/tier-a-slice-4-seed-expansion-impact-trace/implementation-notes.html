Tier A Slice 4 — Seed Expansion in Impact + Trace (Walk Path)

Branch feat/tier-a-name-resolution · 2026-06-06 · GitHub issue #57.

Root Cause Recap

Seam's extractor stores symbol names as qualified strings ("Class.method") but stores call-edge target_name as the bare identifier ("method"). This asymmetry means walk() seeded with ["Parser.parse"] never matches any edge row, because all edges use the bare form as their target. The result: impact("Parser.parse") returned empty upstream before this slice.

Key Design Decisions

Decision expand_impact_seeds in names.py (leaf module)
The new function lives in seam/query/names.py, the existing single source of truth for all qualified<->bare bridging. This keeps impact.py and flows.py thin — they just call this function and pass the results to walk(). The leaf module constraint (names.py imports only stdlib + config) is preserved: expand_impact_seeds reuses is_container_symbol and get_member_names which are already in the same module.
Decision walk() internals unchanged
walk() in traversal.py already handles multi-seed BFS: it starts from all seeds simultaneously at depth 0. Passing ["Parser.parse", "parse"] as seeds makes walk() find callers of either string as edge target_name — exactly what we need, with zero change to traversal.py.
Decision trace() uses target_aliases set for the found check
The BFS in trace() checks if tgt_name in target_aliases (was if tgt_name == target). The aliases set is built from expand_impact_seeds(conn, target) plus the exact target name. When a bare alias is matched (e.g. tgt_name="parse" for target="Parser.parse"), the returned path reflects the actual edge names — it is not renamed to the canonical qualified name, preserving the real hop shape.
Decision Layer import: analysis/ can import from query/
The BACKEND_STRUCTURE layer table explicitly permits analysis/ (non-clustering) to import from query/. Both impact.py and flows.py import expand_impact_seeds from seam.query.names. No circular dependency: names.py imports only stdlib + seam.config.

Expansion Rules (expand_impact_seeds)

InputSeeds returnedWhy
"Parser.parse" (qualified) ["Parser.parse", "parse"] Edges store bare "parse" as target_name; both forms needed
"Parser" (container class) ["Parser", "parse", "validate"] No call edges target a class name directly; fan into members
"orchestrate" (bare, non-container) ["orchestrate"] Exact match — no expansion needed

Tradeoffs

Tradeoff Bare-name impact may over-reach for homonyms
When multiple classes define a method with the same bare name (e.g. both ServiceA.process and ServiceB.process), seeds for "ServiceA.process" include bare "process", which also matches callers of ServiceB.process. This is consistent with how Seam handles the ambiguity (edges are name-keyed, not ID-keyed), and the confidence system labels such hits as AMBIGUOUS. The alternative — no expansion — would return zero results, which is worse.
Tradeoff trace() path shows bare hop, not canonical qualified name
When trace("orchestrate", "Parser.parse") succeeds via bare alias, the returned path shows to_name="parse" (the actual edge target stored), not to_name="Parser.parse". This is honest — it reflects what the extractor stored. Renaming the hop in the result would be misleading about the actual stored graph.
