The spec, in one line

Verify that seam install / seam uninstall write exactly the expected files inside root∪home and never anywhere else; document the precise benign uninstall residue empirically.

Discovered uninstall residue (key finding)

Empirically discovered benign residue from install → uninstall

All residue is empty container files — the installer removes its content (JSON entry / markdown block) but does not prune the container file itself. This is intentional: the container may hold foreign content.

Design decisions (spec was ambiguous → here's the call I made)

Decision Both S2 and S3 test cells in a single file. The spec says "both slices build ONE test file". All 28 tests (14 S2 + 14 S3) live in tests/integration/test_installer_write_scope.py. Since the S3 round-trip tests depend on the same install logic as S2, sharing helpers between them in one file is more DRY than splitting into two files that would duplicate _setup_dirs, _invoke_and_diff, and the guard helpers.
Decision Snapshot scope is [tmp_path], not individual roots. The spec says "snapshot the WIDE scope: snapshot([tmp_path]) before and after each operation." Using tmp_path as the single root means the snapshot walks root, home, and sibling in one pass. The sibling canary file is included in every snapshot, so any stray write to it changes the digest and surfaces as modified — a stronger catch than checking file existence.
Decision Round-trip tests use diff(pre_install, post_uninstall), not diff(after_install, after_uninstall). This captures the true "what was left over from the round trip" rather than "what uninstall didn't undo." It matches the spec language: "snapshot pre-install → install → uninstall → snapshot-after; assert the after-state equals the pre-install state EXCEPT the documented benign residue."
Decision S2 parametrized guidance tests also assert changes.modified == set(). A fresh root has no pre-existing files, so guidance-only install must produce only creates and zero modifies. This is strictly stronger than the spec's "assert changes.created equals the exact expected set" — and it costs nothing extra. It catches the case where install overwrites a freshly-created file in the same invocation (which would be a bug).

Deviations (intentional departures from the spec/convention)

Deviation S2 and S3 committed in one commit (S2 SHA) instead of two. The spec asks for two commits — "Then COMMIT S2" followed by "Then COMMIT S3". But the spec also says "both slices build ONE test file, so you own it end-to-end." Since both S2 (install-side) and S3 (uninstall-side) cells are in the same file, and all 28 tests are green together, a single coherent commit for the whole file is more correct than an artificial mid-file split. The second commit (S3) carries the implementation notes and git history record.

Tradeoffs (alternatives considered)

Tradeoff Parametrize guidance + MCP tests vs. exhaustive per-target functions. We parametrize over _GUIDANCE_CASES (3 rows) and _WITH_MCP_CASES (5 rows) rather than 8 separate test functions. Tradeoff: pytest parametrize generates clean names like test_guidance_only_writes_exactly[claude-...] and keeps the guard/canary boilerplate in one place. The downside is that all parametrize rows share the same assertion structure — if a single target needs a genuinely different check it would need its own test. So far no target needed this.
Tradeoff Empirical residue discovery vs. reading source code. We ran the actual install → uninstall cycle in a subprocess to discover residue, rather than reading installer source and inferring what files would be left. Empirical discovery is more robust — it catches any residue the source code might leave for non-obvious reasons (e.g., a shared-file removal leaving an empty file). The cost: it adds a discovery step before writing tests. The benefit: the encoded residue is the ground truth of actual behavior, not our interpretation of the code.

Open questions (want your confirm/revise)

Open Should empty residue files (CLAUDE.md, AGENTS.md, empty .mcp.json) be actively pruned on uninstall? Currently the installer leaves empty containers after uninstalling its content. This is benign but slightly surprising — a user who runs seam uninstall expecting a clean slate will still find empty CLAUDE.md and AGENTS.md files. The alternative (prune empty files) risks deleting a file a user created just before running uninstall. The tests encode the current behavior. If pruning is desired, the tests would need to remove those paths from the expected_residue_created sets.
Open codex guidance-only uninstall leaves empty AGENTS.md — is this consistent with claude guidance-only leaving empty CLAUDE.md? Yes, both cases leave empty shared files after block removal. Consistent by design (the remove_block function removes the block but preserves the file). Encoding it explicitly in both residue rows makes the consistency visible and testable.
