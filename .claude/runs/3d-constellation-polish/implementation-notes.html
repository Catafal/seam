3D Constellation Polish — Spherical Globe Layout

This is a living document capturing how the implementation interprets, diverges from, or trades off against the spec. Updated as each slice lands.

The spec, in one line

Replace the flat XY-ring+z-smear seed with a deterministic Fibonacci sphere (cluster-grouped for locality); recenter + radial-clamp after FA2; switch node_size from linear to log1p. All pure, deterministic, never-raises, LayoutResult shape unchanged.

S1 — Spherical Globe Layout (#260)

Design decisions (spec was ambiguous → here's the call I made)

Decision Cluster locality via polar-angle banding, not azimuthal proximity. The Fibonacci sphere uses the golden angle (~137.5°) in azimuth specifically to maximise spread — consecutive indices are NOT azimuthally close. What IS preserved between consecutive indices is the polar angle (theta), which varies slowly. By sorting nodes by (fnv1a(cluster_key), fnv1a(name)), same-cluster nodes get contiguous Fibonacci indices and therefore occupy the same polar-angle band (same spherical "cap"). The test for cluster locality checks centroid angular separation rather than pairwise intra-cluster distance, which is more robust for small n.
Decision Depth modulates shell radius (not Z). The spec says "call-depth may modulate shell radius". I implemented this as r = _SPHERE_RADIUS + depth * _DEPTH_RADIUS (30 world units per depth level). Entry points (depth=0) sit on the innermost shell; deeper nodes are on slightly larger shells. The effect is intentionally subtle — it adds a gentle radial stratification without breaking the spherical appearance. This replaces the old z = -depth * 50 smear.
Decision log1p with _SIZE_LOG_FACTOR=1.5 and _SIZE_LOG_CEIL=6.0 for node size. The spec says "sub-linear sqrt/log1p scale with a small base and ceiling, so a hub is only a few× a leaf". I chose log1p over sqrt because log1p grows faster initially then flattens more aggressively — it reaches the ceiling at degree ~50 (realistic for most hubs) while sqrt would still be growing at degree 1000. With these constants, node_size("function", 0)=4.0 and node_size("function", 10000)=10.0 → ratio 2.5×, well within "a few×".
Decision Expose _sphere_seed_positions and _recenter_and_clamp as importable private helpers for TDD. The spec asks for tests on the sphere shell, cluster locality, and outlier clamping. Testing these properties through compute_layout is hard because FA2 modifies positions. Exposing the two helper functions with a leading underscore (consistent with _fnv1a which was already exposed) allows precise unit tests without the full pipeline. The functions are pure and stateless — exposing them adds no coupling risk.

Deviations (intentional departures from the spec/convention)

Deviation No per-node jitter in the sphere seed. The original ring seed added per-node LCG jitter (±40 world units in X/Y) to break symmetry between nodes with the same cluster key. The Fibonacci sphere does not need jitter: the within-cluster ordering by FNV-1a(name) already gives each node a unique polar index, and the golden-angle azimuth step ensures no two nodes ever overlap. Adding jitter to a sphere seed would move nodes off the shell, complicating the shell-radius test. Removing it is safe and correct.
Deviation Old constants (_RING_MIN, _RING_SPAN, _JITTER, _DEPTH_Z) removed entirely. The spec says "replace" the flat ring seed. Rather than keeping the old constants as dead code, I removed them and replaced with the sphere constants (_SPHERE_RADIUS, _DEPTH_RADIUS, _PHI_GOLDEN, _OUTLIER_K, _SIZE_LOG_FACTOR, _SIZE_LOG_CEIL). No code outside layout.py referenced these constants — confirmed by search. This keeps the module clean and prevents future confusion about which seed path is active.

Tradeoffs (alternatives considered)

Tradeoff mean+k·sigma clamping vs. MAD-based or percentile clamping. The spec specifies "mean + k·sigma". This works well for n≥20 with a small fraction of outliers, which is the realistic FA2 output scenario (the anchor spring keeps most nodes bounded). For very small n (n=7) with a single extreme outlier, mean+k·sigma fails because the outlier inflates sigma enough to raise the clamp threshold above its own radius. I designed the clamp tests to use n=50 normal nodes + 1 outlier (a realistic ratio), which the formula handles correctly. The test documents this n-dependency in the function's docstring. A MAD-based estimator would be more robust but departs from the spec.
Tradeoff sqrt vs. log1p for node_size sub-linear scaling. sqrt(degree) is the other natural sub-linear scale. At degree=100, sqrt gives 10 (needs a factor and ceiling to match the spec's "a few×"); at degree=10000, sqrt gives 100 (still needs capping). log1p reaches its ceiling much earlier (effectively saturates at degree~50 given our constants) and the resulting size curve is visually smoother. Both are valid; log1p was chosen because it eliminates the visible kink at the ceiling that sqrt-with-hard-cap would have.

Open questions (want your confirm/revise)

Open _DEPTH_RADIUS = 30 world units per depth level — correct magnitude? With _SPHERE_RADIUS=600 and _DEPTH_RADIUS=30, a depth-5 node sits at radius 750 (25% larger shell). In a typical codebase with max depth 10–15, the outermost shell reaches 900–1050 (50–75% larger). Whether this creates a visible "globe within a globe" effect or is too subtle to notice depends on the actual depth distribution. The constant is easily tunable in layout.py without a re-index. Ran with by assumption; confirm during visual review of the rendered constellation.

S2 — Additive node glows, color by KIND (#261)

Design decisions (S2)

Decision Chose "additive InstancedMesh" (documented fallback) over Points. The spec offered two approaches: (a) Points with raycast threshold, or (b) the documented fallback — an additive emissive InstancedMesh with depthWrite=false. I chose (b) for two reasons: (1) InstancedMesh raycasting for hover/click was already working correctly — switching to Points would require re-implementing threshold-based raycasting and re-testing. (2) The visual result is identical: AdditiveBlending + depthWrite=false + transparent on the existing sphereGeometry turns every sphere into an additive glow that never punches a black hole. Raycasting on the sphere mesh is more reliable than threshold-based point picking, especially for small nodes far from camera.
Decision KIND_COLORS is the single source of truth for node hue; node.color is ignored. The stellar/degree color in node.color (from the layout engine) encoded degree in hue, making the visual meaning ambiguous (is this a bright node because it's a hub or because it's a blue-giant class?). After S2, hue = kind (via KIND_COLORS[node.label] ?? DEFAULT_KIND_COLOR) and degree = size + brightness (the highlight boost). A new exported kindColor(label) pure function is the single resolution point — the filter legend and the node renderer both derive from it, so they can never drift apart. The test suite enforces this explicitly with a "same source of truth as the legend" assertion.
Decision Highlight boost logic kept identical, applied to kind color instead of stellar color. The boost = 1.2 + brightness × 0.8 formula from S1 was designed for stellar colors (reddish dwarfs to white giants) but works equally well for kind colors: purple (class) and cyan (function) have enough channel weight that boost×channel exceeds 1.0 on the dominant channel, triggering Bloom. The test confirms this by asserting r + b > 2.0 for a highlighted class node (purple has high r and b). No tuning needed — the formula is channel-agnostic.

Deviations (S2)

Deviation sphereGeometry kept (not replaced with a billboard sprite or plane). The spec says "soft point-glows" but doesn't mandate the geometry primitive. With AdditiveBlending, spheres behave as additive glows on the dark background — they are indistinguishable from sprites at typical camera distances. Keeping sphereGeometry avoids re-implementing the raycasting surface (plane quads don't raycast well at oblique angles, and Points need a threshold). The visual outcome (additive, non-occluding glows) is fully met.

S3 — Calmer edges + controlled bloom (#262)

Design decisions (S3)

Decision Named constants for every intensity level (INTENSITY_HL_BOTH, INTENSITY_SAME_CLUSTER, LOUD_KIND_DIM, etc.) instead of bare literals. The original code had bare 0.25 and 0.06 in the middle of the loop with no indication of what they represented. Extracting them to module-level constants with descriptive names makes the intent auditable, makes it easy to tune later, and makes the tests anchor to readable names rather than magic floats. Zero runtime cost — constants are inlined at build time.
Decision Loud-kind dim applied via a multiplicative factor (0.5×) on the base intensity, not by per-channel clamping or a separate color table. The spec says "reduce visual weight of loudest non-call kinds in the NO-HIGHLIGHT field; EDGE_TYPE_COLORS semantics stay". The cleanest factoring is: intensity = base × kindMult, where kindMult = LOUD_KIND_DIM for loud kinds and 1.0 for all others. This leaves the color pipeline unchanged (still reads EDGE_TYPE_COLORS[e.type]), preserves the ratio of channels within each kind's color, and is trivially reversible. Chosen 0.5 because it brings instantiates maxchannel (R=0.976×0.05=0.0488) below call maxchannel (G=0.635×0.10=0.0635) while keeping both kinds visible.
Decision Loud kinds NOT dimmed when highlighted — the dim is strictly no-highlight-path-only. The spec says "full color returns when endpoints are part of a highlighted selection." Implementing this by only entering the LOUD_KIND_DIM path in the else (no-highlight) branch satisfies the spec without any additional conditional. The test anchors this: instHL[0] > 0.4 (0.976×0.5=0.488 without dim; 0.244 with — threshold 0.4 distinguishes).
Decision New intensity constants: same-cluster 0.25→0.10, cross-cluster 0.06→0.02. The goal is "calmer ambient field". The ratio same/cross is preserved (5× vs 4.2× before — slightly wider spread, which is fine). The absolute values were chosen so: (1) same-cluster call edges are clearly visible (G=0.0635) but not overwhelming on the teal background, and (2) cross-cluster edges (G=0.0127) are barely perceptible "connective tissue". These numbers can be tuned further without test changes because tests assert only the ORDERING properties, not exact values (except the <0.159 upper-bound test which remains satisfied at 0.0635).
Decision Bloom threshold 0.3→0.6, intensity 1.2→0.8, radius 0.6→0.65. The white-out problem on dense hub clusters is caused by the low threshold (0.3) causing medium-intensity node pixels to bloom, and the high intensity (1.2) spreading that bloom aggressively. With threshold=0.6: ambient edges (max ~0.0635) and even highlighted edges (max ~0.488 for instantiates R) sit below the threshold, so ONLY the node cores (boosted above 1.0 by NodeCloud's highlight path) bloom. The slightly higher radius (0.65) compensates for the reduced intensity so hubs still have a visible corona. These are visual values — not unit-tested. Validated by reasoning about the channel arithmetic above.

Tradeoffs (S3)

Tradeoff 0.5× loud-kind dim vs. per-kind table with individual multipliers. A full per-kind table (e.g. instantiates→0.4, writes→0.45, uses→0.5) would give finer control but requires maintaining a second mapping in parallel with EDGE_TYPE_COLORS. A single LOUD_KIND_DIM=0.5 applied to a set of three kinds is simpler and achieves the visual goal: all three warm kinds drop below call in the ambient field. If one kind proves to need a different multiplier, the LOUD_KIND_DIM constant can be split per-kind then. The chosen value also happens to be easy to understand: "warm edges appear at half the brightness of structured edges".
Tradeoff Bloom threshold 0.6 vs. even higher (e.g. 0.75) to ensure highlighted edges never bloom. At 0.75 the bloom contract would be watertight for all edge intensities (max=0.488 < 0.75 with headroom). But 0.75 might also suppress bloom on node cores that are only moderately boosted (e.g. a low-degree function node: KIND_COLORS.function G=0.702 × boost_factor). The chosen 0.6 leaves a clear gap above the max-edge value (0.488) and well below what a boosted node core reaches. If future slices change the boost formula, the threshold may need re-tuning.

S4 — Click isolates, never navigates (#263)

Design decisions (S4)

Decision Remove onFocusSymbol entirely from ConstellationTab props — not just App.tsx. The spec required removing setCenterSymbol(name) from the onFocusSymbol callback in App.tsx. I went one step further: the prop is deleted from ConstellationTabProps entirely, and the corresponding call is removed from handleSelect. This makes the isolation contract structural — no outbound navigation callback can ever exist on the component. The TypeScript type system enforces it: if someone re-adds onFocusSymbol, the regression test (expectTypeOf<Props>().not.toHaveProperty("onFocusSymbol")) and the tsc compile step both fail immediately. This is a stronger guarantee than patching the App.tsx callback in isolation.
Decision Inbound 2D→3D sync simplified after removing the round-trip guard. Previously, ConstellationTab.useEffect had a round-trip guard: if (focusSymbol === selectedNode?.name) skip, designed to prevent the 3D scene from re-flying when it itself was the source of the focusSymbol change (via onFocusSymbol → App.setFocusSymbol → prop update). After removing onFocusSymbol, there is no round-trip: the 3D tab never writes to focusSymbol in App, so the round-trip cannot occur. The guard was removed; the simpler dedup is just "skip if focusSymbol === lastFocused.current" (unchanged-value guard). Fewer moving parts, clearer intent.
Decision Canvas background click deselect via R3F onPointerMissed — not a transparent overlay mesh. Two options for "click empty canvas to deselect": (a) add a full-viewport transparent mesh that catches all misses, or (b) use R3F's Canvas.onPointerMissed which fires when a pointer event exits the scene without hitting any object. Option (b) is a first-class R3F API — cleaner, no extra draw call, and correct: it fires only on genuine misses (click with no raycast hit), NOT on orbit-drag releases (which are pointer-up events after a pointermove). This means orbiting the scene never accidentally deselects the current node. onPointerMissed on <Canvas> is the right choice.
Decision Esc key listener on document (not the Canvas element). The R3F canvas does not natively propagate keyboard events (it is a WebGL canvas element; there is no key focus model). Attaching the listener to document means Esc fires regardless of which DOM element has focus — consistent with how other tooling (VS Code, command palettes) handles escape-to-dismiss. The listener is cleaned up on ConstellationTab unmount. A ref to handleClose is not needed because handleClose itself is a useCallback with stable identity (no deps).
Decision prefersReducedMotion() called once at mount via useRef, not reactively. The prefers-reduced-motion media query could be listened to reactively with addEventListener("change"). In practice, users do not change this setting during a page visit. Calling it once per component mount (via useRef(prefersReducedMotion())) avoids a listener + effect pattern that would be over-engineered for this use case. The ref makes the value stable across frames (no re-render needed; useFrame reads the ref synchronously).
Decision Reduced motion: snap camera in ONE frame, not immediate DOM manipulation. When prefersReducedMotion = true, CameraAnimator skips the lerp and uses camera.position.copy(target) + camera.lookAt(target.lookAt) in a single useFrame call, then marks progress.current = 1.0 to prevent further updates. This is one frame of visual jump, not a zero-frame snap (which would require breaking the R3F animation loop contract). One frame is imperceptible and the correct minimal change inside the animation loop.
Decision Discoverability hint is a static HTML overlay in ConstellationTab (not inside R3F). The hint ("Click a node to isolate its connections · click empty space or press Esc to reset") is a <p> positioned absolute bottom-4 over the canvas, outside the R3F tree. Inside R3F it would require <Html> from drei and world-space positioning. The hint is a UI affordance, not a 3D element — HTML is the right tool. It is pointer-events-none so it never blocks canvas interaction. Hidden once a node is selected so the NodeDetailPanel has clean visual space.

Deviations (S4)

Deviation Dropped the inbound focusSymbol→handleNavigate indirection; now calls handleSelect directly. The spec says "the inbound App→3D focus sync — flying to a symbol picked in the 2D side — may stay". It stayed. But previously the sync went through handleNavigate → handleSelect, where handleNavigate found the node by name and called handleSelect. Since the round-trip guard in the effect was simplified (see decision above), I inlined the node lookup directly into the effect and call handleSelect(node). One fewer level of indirection, same behavior.

Tradeoffs (S4)

Tradeoff Type-level regression test (expectTypeOf) vs. integration test. The ideal regression test would render App in 3D mode, simulate a node click, and assert that centerSymbol / mode did not change. That test requires rendering R3F in jsdom (which requires mocking the WebGL context, canvas, and the full drei/fiber call chain) — complex, fragile, and not worth the maintenance cost for what is essentially a "this callback doesn't exist" proof. The chosen approach — expectTypeOf<Props>().not.toHaveProperty("onFocusSymbol") — provides a compile-time guarantee enforced by both vitest and tsc --noEmit. If someone adds the prop back, the test fails. If someone passes a navigation callback via a different prop name, a code review catches it. The type assertion is the right level of proof here.
