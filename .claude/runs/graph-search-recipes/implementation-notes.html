The spec, in one line

Add a transparent, deterministic recipe layer over existing typed graph-search so agents can choose safe structural queries by intent, with caveats, follow-up calls, transport parity, docs, and benchmark coverage.

Design decisions (spec was ambiguous → here's the call I made)

Decision Start with one deep catalog module. Recipes will compile into existing graph-search parameters before any DB work happens, because the PRD explicitly wants recipes to be product logic rather than a second graph engine.
Decision Recipe overrides are inferred from non-default graph-search arguments. Existing CLI/MCP signatures pass default values eagerly, so recipe compilation treats graph-search's generic defaults as omitted values and records only non-default values as overrides.

Deviations (intentional departures from the spec/convention)

Deviation No material deviations. The implementation follows the PRD boundary: no schema migration, no new extraction, no LLM planner, no arbitrary graph language. Workspace graph-search was included in the parity slice because it already exposes the same typed graph-search surface.

Tradeoffs (alternatives considered)

Tradeoff Recipe ids vs. broader natural-language parsing. Stable ids are less magical, but they are deterministic, testable, MCP-friendly, and preserve Seam's local read-only trust boundary.

Open questions (want your confirm/revise)

Open None blocking. Future catalog growth should be driven by recurring answerability scenarios, not by making the first recipe set exhaustive.
